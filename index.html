Barry Schoedel | Learning Designer and Technologist

1) LearnCAT: Diocese of Baton Rouge

Project: Create a learning environment that faciliates Catholic faith formation.
I architected, configured, and deployed the LearnCAT Learning Management System for the Diocese of Baton Rouge. By designing a centralized, easily accessible digital infrastructure, the system renewed formation efforts and allowed for scalable, asynchronous faith formation and administrative training across diverse parish communities and Catholic schools.

This project included the development and implementation of a new eight course curriculum. I employed 5 diocesan instructors to help with course development as subject matter experts. I was the content creator for, The Catholic Identity of the Catholic School course, and the editor for all the others.

Applied Adult Learning Principle: Social Cognition and Communities of Practice: Situated cognition (or social cognition) holds that learning is inseparable from the context in which it occurs. Lave & Wenger have made an enormous contribution to organizational culture in the United States and Canada through the development of our conception of communities of practice. The real gift of their research is that it helped people recognize that learning isn't an individual cognitive exercise.

Adult professionals who participate in communities of practice benefit from the culture of that community in ways that deepen their knowledge and professionalization. I differ from Lave and Wenger in that I am not a strict constructionist but I value their insight. In learning design, especially in particular contexts, I consider how I can facilitate the development of communities of practice (Lave & Wenger, 1991).

This particular design used a blended format with xAPI files exported from Rise for learning content sections.The design has seven stages: 1) Course overview, which is like a syllabus but also includes information about how to use the platform 2) Introductory Exercise - this helps learners start to use the social features of the learning platform and get to know each other (using the forum) 3) Learning Content sections - opportunity for interested adults to grow in knowledge about the various topics 4) Instructor Faciliated Section: cohort is together with a facilitator fielding questions about content and bringing up challenging issues about it. By drawing on the knowledge the cohort already has this kind of facilitation can help professionalize as each learner is particpating and able to contribute - it's not a didactic session.

This pattern is repated with 5) Learning Content II 6) Instructor Facilitated II 7) Concluding Exercise. Using a cohort model and encouraging participants to grow as communities of practice makes this learning experience different than one that is merely didactic. There are elements to the learning content that are more didactic but that is only a small part of the larger course design.

Technical design: This platform was built using the open source version of WordPress, extending it with an LMS called Learndash. The course content was developed using Articulate and those files were exported/imported as xAPI (rather than SCORM). The choice for xAPI was for deeper learner analytics and the relative drag of SCORM files. LearnCAT also utilizes WooCommerce for payment/registration and has a security extension that includes MFA and DDoS protection. This and various forms are automated to send data to Google Sheets.

2) Cyril-OCIA Training Chatbot

Project: Create a learning object that is adaptive for an OCIA formation course.

I engineered a specialized conversational AI agent to support the Order of Christian Initiation of Adults (OCIA). The chatbot was explicitly designed to act as an educational tool that facilitates knowledge growth and practice confidence for OCIA leaders and catechists

Adult Learning Principle: Scaffolded Practice & Andragogy: While andragogy has been challenged in different ways, Scholars demonstrate that Knowlesian andragogical principles are still considered widely applicable. Studies have demonstrated that their use yields better results than merely didactic approaches.

The chatbot's conversational architecture provides graduated, responsive scaffolding, an approach that can adapt to the level of understanding of the learner and provide support based on that foundation. There is a significant body of literature that supports scaffolding for young and adult learners.

Cyril wasn't built to be a standalone course but rather a resource within a course to use as a knowledge base but also to help make drafts of OCIA pastoral plans based on the literature it has access to. The larger course has only been prototyped but methodologically it facilitates pastoral practice away from the screen rather than learning in a way where learners are not in the actual reality where OCIA takes place. All of my course design considers the benefits and dangers of using screens in the learning process. I find creative ways to faciliate learning activities that take place off screen and unlike some profit based tools don't seek to keep users dependent upon the screen. There are many tools for keeping learners engaged on screen, but as Catholic we have a responsibility to orient people towards real relationships. This personalist philosophy is part of my design approach.

Technical Design: Cyril is built on a scalable Angular 21 framework, utilizing Angular Material and Tailwind CSS/PostCSS for a modern, responsive UI. The core learner experience is powered by the Google Gemini API, with provider interchangeability. A key feature is the RAG-style knowledge base, which leverages Firebase Firestore to enable administrators to update training data dynamically from within the application. Geospatial capabilities are integrated via Leaflet.js, facilitating interactive parish mapping.

Data Orchestration & Deployment: The system streamlines administrative tasks through automated workflows connecting Firebase, SendGrid, Google Sheets, and various Google APIs. The deployment architecture has matured from an initial Cloud Run prototype to a robust production environment on a Virtual Private Server (VPS). A custom build automation script handles the orchestration of the full stack: 1) Landing Page: React-based frontend. 2) Chat Interface: Angular application. 3) Backend: Express API, managed via PM2 to ensure persistent performance and high availability.

3) DRM – Diocesan Relationship Manager

Project: Assist diocesan personnel in developing a consistent source of truth for those they serve to more effectively carry out the mission of their respective areas.

Developed a Diocesan Relationship Manager to track relations with parish ministry personnel, relevant school personnel, peers from other dioceses, apostolate leaders and vendors who work with the diocese.

While this platform wasn't designed as a course, learning activity or learner experience it functions as a means of managing knowledge in the form of vital information. Doing this well help personnel learn who their constituents and peers, etc. are across different areas of ministry. This is especially helpful for addressing knowledge loss from turnover and from knowledge silos from office to office.

Adult Learning Principle: Connectivism (Siemens, 2005). In a digital age, learning occurs not just within the individual, but across the networks and relationships that individuals maintain within their organization.

Connectivism argues that in networked digital environments, learning resides in the connections between nodes — people, resources, and organizations — and that the capacity to maintain and traverse those networks is itself a core competency. Utilizing these connected networks becomes integral to learning.

Connectivism is a contemporary theory of learning that can be applied to professional settings that are highly dependent upon connected organizational structures (nodes) for success.Research shows that connectivist principles provide a coherent architecture for leadership and organizational development in digital contexts. In addition, empirical research demonstrates that connectivism-informed design strengthens learners' capacity for adaptive, networked knowledge construction in complex institutional environments.

Technical design: ● The DRM (Diocesan Relationship Manager) is a web-based information system designed to model the organizational, pastoral, and administrative complexity of a Catholic diocese. Technically, it combines a Laravel backend with a modern JavaScript frontend to manage structured entities such as contacts, clergy, parishes, organizations, forms, calendars, and custom data fields. Its architecture supports both internal administrative workflows and public-facing data collection through configurable forms, while maintaining relational links between people, roles, institutions, and ministry activities. This makes it possible to move beyond isolated spreadsheets or basic CRM records and instead manage diocesan life as an interconnected system.

From an engineering standpoint, the platform is built for extensibility and operational visibility. It uses database-backed models and migrations to represent diocesan structures, supports authenticated staff workflows, and includes configurable entity types and custom fields so the data model can evolve without redesigning the application from scratch. Additional features such as liturgical calendar integration, workflow-oriented dashboards, and structured form submissions position the DRM not just as a contact manager, but as a diocesan operations platform capable of supporting communication, planning, reporting, and long-term institutional knowledge.

4) xAPI LRS and Statement Monitor

Project: Develop a tool for monitoring learner progress in a way that gives insight into learner behavior as well as tracking errors in the LMS

Built a backend tracking system utilizing the Experience API (xAPI) to capture granular, cross-platform learning behaviors. Moving beyond standard SCORM completion tracking, this monitor gathered data on engagement, time-on-task, and interactive choices to evaluate instructional effectiveness.

Applied Adult Learning Principle: Data-Driven Metacognition (Winne & Baker, 2013). By analyzing behavioral patterns, instructional designers can adjust learning paths in real-time to match the learner's actual cognitive state rather than a presumed pace.

The purpose of using the xAPI file type instead of SCORM was to better understand and respond to learner activity. This can facilitate learning for the learner but also give administrators vital information about learner behavior in particular learning environments.

Data-driven metacognition can be understood in a different ways, e.g. when adult learners realize there is some awareness of and accountability for their activity in a learning environment they are more likely to reflect upon and adjust how they think about it in ways that influences behavior positively for learning.

However, when the system doesn't provide those insights due to an error and/or gets in the way of learner progression, it has a negative impact. When utilizing technologies to facilitate learning, a big part of that can be addressing bugs and shortcomings of these various technologies. This plugin I developed is one tool to do that.

Evidence-Based Design: xAPI (Experience API) packages generate structured learning event statements that record user activity in the form of data, often with timestamps, results, context, and completion metadata. Unlike SCORM, which is tightly coupled to an LMS runtime, xAPI statements can be sent from multiple environments and learning experiences, but they require a Learning Record Store (LRS) to receive, validate, store, and query those statements for analytics. The LRS is what makes xAPI useful at scale: it serves as the system of record for engagement, progression, completion, and behavioral data, enabling reporting across learning modules, platforms, and user contexts rather than confining analytics to a single course shell.

The xAPI Statement Monitor was developed as a diagnostic layer to identify breakdowns between xAPI content, the LRS, and LearnDash course progression logic. In practice, it was used to trace malformed or missing statements, monitor whether expected completion and progress events were actually being transmitted, and compare those events against LearnDash’s own progression and completion states inside WordPress. This made it possible to detect where learner analytics and LMS completion behavior were diverging, isolate issues in statement formatting or delivery, and support more reliable reporting on whether learners had actually progressed through course activities as intended.

5) LXPython – Mobile-First Instruction Environment

Project: Design a mobile learning environment for a programming language that wasn't immediately boring, that doesn't start the same way every time, that was challenging and effective.

Designed and deployed LXPython, a mobile-first learning application. This project integrated official technical documentation with agentic AI facilitation, allowing adult learners to engage with complex programming material through a structured, highly modular environment.

LXPython, a mobile-first learning application for learning the Python programming language. I intentionally designed it for adult ADHD learners using evidence-based approaches. It starts out difficult rather than with the most basic and trivial information. It is project-based and structured for informal learning rather than a traditional school or text-book based approach. It is designed for people who want to learn rather than for people for whom learning is compulsory.

This project integrated official technical documentation with agentic AI facilitation, allowing adult learners to engage with complex programming material through a structured, modular, project-based, environment.

Applied Adult Learning Principle: Self-Directed Learning (Knowles, 1975).Adults seek informal learning environments and approaches for personal and professional growth. Much of the learning that adults engage in is self-directed. When self-motivated adults prefer to be challenged in a way that doesn't waste their time.

Technical Design LXPython is technically designed as an Angular 21 learning application with a modern TypeScript frontend, Angular Material/CDK UI primitives, Tailwind/PostCSS styling, and optional server-side rendering through Angular SSR with a small Express host. Its core interactive feature is an in-browser Python runtime powered by Pyodide, which lets learners execute Python code directly in the client without depending on a remote execution server. The interface also embeds Monaco Editor for structured coding exercises, which makes the platform feel closer to a lightweight IDE than a static course page.

On the data and intelligence side, LXPython uses Firebase Authentication and Cloud Firestore to persist learner profiles, chat history, learning progress, memories, code attempts, forum posts, and user secrets. The teaching layer is implemented as an AI learning agent service that builds responses from user profile data and stored learning history, then calls an LLM API to generate calibrated instructional guidance. In practice, that means the system is architected as a stateful adaptive learning environment: browser-based code execution for immediate practice, Firestore-backed learner state for persistence, and an AI facilitation layer that personalizes explanations, feedback, and progression over time.
